简介

什么是 MyBatis？

  • MyBatis 是一款优秀的持久层框架，它支持定制化 SQL、存储过程以及高级映射。MyBatis 避免了几乎所有的 JDBC 代码和手动设置参数以及获取结果集。MyBatis 可以使用简单的 XML 或注解来配置和映射原生信息，将接口和 Java 的 POJOs(Plain Ordinary Java Object,普通的 Java对象)映射成数据库中的记录。MyBatis 官方中文文档

环境搭建与入门使用

如果是使用 Maven 工程：

如果是普通工程：MyBatis下载地址

  1. 导包；

    1. mybatis-3.5.5.jar ；
    2. mysql-connector-java-5.1.0-bin.jar：数据库驱动；

  2. 写配置；

    1. 第一个配置文件（**全局配置文件**）：MyBatis 全局配置文件，指导 MyBatis 如何正确的运行，比如连接哪个数据库；

    2. 第二个配置文件（**映射文件）：编写每一个方法该如何向数据库发送 sql 语句，如何执行等等（该配置文件相当于接口的实现类**，一个配置文件对应一个接口）

      我们写的 dao 接口的实现文件，MyBatis 默认是不知道的，需要在全局配置文件中注册;

  3. 测试；

    1. 根据全局配置文件先创建一个 SqlSessionFactory；
    2. 从 SqlSessionFactory 对象中获取 SqlSession 对象；
    3. 然后使用 SqlSession 获取接口的实现类，然后操作数据库；

补充事项

  1. 获取 sqlSession.getMapper(Class clazz) 获取到的是是一个代理对象，MyBatis 自动创建的；
  2. SqlSessionFactory 和 SqlSession 对象；
    1. SqlSessionFactory 创建 SqlSession 对象，Factory 只 new 一次即可；
    2. SqlSession 相当于 Connection 与数据库进行交互的连接，每和数据库进行一次会话，就应该创建一个新的 SqlSession 对象。

全局配置文件

全局配置文件：指导 MyBatis 正确运行的一些全局设置。

官方文档详细且易懂：官方文档

Configuration配置

  • properties 属性：类似于 Spring 的 context:property-placehoder，作用是引用外部配置文件 ；

    • resource：从类路径下开始引用；
    • url：引用磁盘路径或者网络路径的资源；
    • 例如：用 properties 引用外部数据库连接配置，然后就可以直接在 environments -> environment -> dataSource -> property 标签使用${}来动态取值；

  • settings 设置：这是 MyBatis 中极为重要的调整设置，它们会改变 MyBatis 的运行时行为；

  • typeAliases 类型命名：类型别名可为 java 类型设置一个缩写名字。它仅用于 XML 配置，意在降低冗余的全限定类名书写；

  • typeHandlers 类型处理器：MyBatis 在设置预处理语句（PreparedStatement）中的参数或从结果集中取出一个值时，都会用类型处理器将获取到的值以合适的方式转换成 Java 类型；

  • objectFactory 对象工厂：每次 MyBatis 创建结果对象的新实例时，它都会使用一个对象工厂（ObjectFactory）实例来完成实例化工作；

  • plugins 插件：MyBatis 允许你在映射语句执行过程中的某一点进行拦截调用。默认情况下，MyBatis 允许使用插件来拦截的方法调用包括：

    • Executor (update, query, flushStatements, commit, rollback, getTransaction, close, isClosed)；
    • ParameterHandler (getParameterObject, setParameters)；
    • ResultSetHandler (handleResultSets, handleOutputParameters)；
    • StatementHandler (prepare, parameterize, batch, update, query)；

  • environments 环境；

    • environment 环境变量；
      • transactionManager 事务管理器；
      • dataSource 数据源；

  • databaseldProvider 数据库厂商标识：考虑数据库移植的；

  • mappers 映射器；

    • url：引用磁盘路径或者网络路径的资源；
    • resource：在类路径下找sql映射文件；
    • class：直接引用接口的全类名；
      • 可以将 xml 放在 dao 接口的同目录下，而且要文件名和接口名一致；
      • 也可以 class 的另一个用法使用注解，直接在接口的方法上加注解；
    • 建议：配合使用，重要的或复杂的 dao 可以些配置文件，简单的 dao 就直接注解；

参数处理

#{key}取值的时候，也可以进行一些规则设置；

  • 参数位置支持的属性：javaType、jdbcType、mode、numericScale、
    resultMap、typeHandler、jdbcTypeName、expression；
  • 实际上通常被设置的是：可能为空的列名指定 jdbcType；
  • 默认不指定jdbcTyep：mysql和Oracle都没问题；
    • 但万一传入的值是 null，mysql 插入 null 值没问题，oracle 会有问题，因为 Oracle 不知道 null 是什么类型；

#{key}：获取参数的值，预编译到 SQL 中，安全；

${key}：获取参数的值，拼接到 SQL 中，有 SQL 注入问题。ORDER BY $(name}；

sql语句只有参数位置是支持预编译，当表之类的需要动态指定的时候，就使用${}；

参数的各种取值

单个参数：

  1. 基本类型：取值 #{随便写} 随便写一个字符串都能获取到单个参数；
  2. 传入了 JavaBean：取值 #{javaBean的属性名} ；
  3. 传入了 Map：将多个要使用的参数封装起来；
    • 取值 #{封装 map 时用的 key}。

多个参数：

  • 取值时用 #{参数名} 取值是无效的，用参数索引却是可以的，例如 #{0}、#{1} 或者使用 param1~paramN，例如：#{param1}、#{param2}；
  • 原因：只要传入了多个参数，MyBatis 会自动的将这些参数封装在一个 Map 中，封装使用的 key 就是参数的索引和参数的第几个表示。所以，*#{key} 就是从这个 map 中取值*。

命名参数：我们可以用 @Param 来指定封装参数时用什么作为 key

  • 例如：Employee getEmpByIdAndEmpName(@Param("id")Integer id, @Param("empName")String empName);
    • 取值：#{@Param 注解里的 value 值} ；

扩展：多个参数，自动封装为 map；

例如：method1(@Param("id")Integer id, String empName, Employee employee);

  • 要取id的值：#{id} ；
  • 要取 empName 的值: #{1} 或 #{param2} ；
  • 要取 employee 里的 email 属性值：${param3.email} 。

id & result 标签

这些元素是结果映射的基础。id 和 result 元素都将一个列的值映射到一个简单数据类型（String, int, double, Date 等）的属性或字段。

这两者之间的唯一不同是，id 元素对应的属性会被标记为对象的标识符，在比较对象实例时使用。

Id 和 Result 的属性说明：

  • property：映射到列结果的字段或属性。如果 JavaBean 有这个名字的属性（property），会先使用该属性。否则 MyBatis 将会寻找给定名称的字段（field）；无论是哪一种情形，你都可以使用常见的点式分隔形式进行复杂属性导航，如：“username” 或 “address.street.number”；
  • column：数据库中的列名，或者是列的别名。一般情况下，这和传递给 resultSet.getString(columnName) 方法的参数一样。

更多属性说明请参考官方文档。

association 标签

关联（association）元素处理 “有一个” 类型的关系。

关联结果映射和其它类型的映射工作方式差不多。你需要指定目标属性名以及属性的 javaType (大多数情况下，MyBatis 都能自己推断出来)。

关联的不同之处是，你需要告诉 MyBatis 如何加载关联。MyBatis 有两种不同的方式加载关联：

  • 嵌套 Select 查询：通过执行另外一个 SQL 映射语句来加载期望的复杂类型；
  • 嵌套结果映射：使用嵌套的结果映射来处理连接结果的重复子集。

association 标签的属性和普通的结果映射相比基本一致（property、javaType等），区别在于 select 和 resultMap 属性。

  • 你执行了一个单独的 SQL 语句来获取结果的一个列表（就是“+1”）；

Mybatis 一个解决办法就是，采用查询延迟加载。将大量语句同时运行的开销分散。另一个办法就是嵌套结果映射。

集合元素和关联元素几乎是一样的。

集合的嵌套 Select 查询

coleection 与 association 的不同，有一个 ofType 属性。这个属性非常重要，它用来将 JavaBean（或字段）属性的类型和集合存储的类型区分开来。例如：

  • javaType 对应的 ArrayList 就是集合的类型；
  • ofType 对应的 Post 就是 集合存储的类型，即 Bean类型。

在一般情况下，MyBatis 可以推断 javaType 属性，因此并不需要填写。所以很多时候你可以简略。

集合的嵌套结果映射

基本上，collection 与 association 除了新增的 “ofType” 属性，它和关联的完全相同。

集合的多结果集（ResultSet）

基本上，collection 与 association 除了新增的 “ofType” 属性，它和关联的完全相同。

示例可以参考官方文档。 （别说我偷懒~🐶）

对关联或集合的映射，并没有深度、广度或组合上的要求。但在映射时要留意性能问题

自动映射与高级映射

正如前面所介绍的，在简单的场景下，MyBatis 可以为你自动映射查询结果。但如果遇到复杂的场景，你需要自定义构建一个结果映射。 同时，我们可以对其进行混合使用。

  • 当自动映射查询结果时，MyBatis 会获取结果中返回的列名并在 Java 类中查找相同名字的属性（忽略大小写）。 这意味着如果发现了 ID 列和 id 属性，MyBatis 会将列 ID 的值赋给 id 属性；
  • 通常数据库列使用大写字母组成的单词命名，单词间用下划线分隔；而 Java 属性一般遵循驼峰命名法约定。为了在这两种命名方式之间启用自动映射，需要将 mapUnderscoreToCamelCase 设置为 true；
  • 甚至在提供了结果映射后，自动映射也能工作。在这种情况下，对于每一个结果映射，在 ResultSet 出现的列，如果没有设置手动映射，将被自动映射。在自动映射处理完毕后，再处理手动映射。

自动映射有三种等级：

  • NONE - 禁用自动映射。仅对手动映射的属性进行映射。
  • PARTIAL - 对除在内部定义了嵌套结果映射（也就是连接的属性）以外的属性进行映射
  • FULL - 自动映射所有属性。

默认值是 PARTIAL，这是有原因的。当对连接查询的结果使用 FULL 时，连接查询会在同一行中获取多个不同实体的数据，因此可能导致非预期的映射。

缓存机制

缓存：暂时存储一个数据；加载系统的查询速度等等；

MyBatis 内置了一个强大的事务性查询缓存机制，它可以非常方便地配置和定制；

一级缓存：线程级别的缓存；本地缓存；SqlSession 级别的缓存；

二级缓存：全局范围的缓存；出除过当前线程；SqlSession 能用外，其它也可以使用。

一级缓存

默认情况下，只启用了本地的会话缓存，它仅仅对一个会话中的数据进行缓存。

机制：只要之前查询过的数据，MyBatis 就会保存在一个缓存中（Map）；下次获取就直接从缓存中拿。会话期间只要查询过的数据都会保存在当前 SqlSession 的一个 Map 中，保存key的格式：**hashCode + 查询的Sql的Id + 编写的sql语句 + 参数**

一级缓存失效的情况：

  1. 不同的 SqlSession，使用了不同的一级缓存。因为一级缓存是 SqlSesion 级别的;
    • 只有在同一个 SqlSession 期间查询到的数据会保存在这个 SqlSession 的缓存中，下次使这个 SqlSession 获取同样的数据，就会从缓存中拿；
  2. 同一个方法不同的参数，由于可能之前没查询过，所以还会发新的 sql；
  3. 在 SqlSesion 期间执行任何增删改操作，就会清空缓存；
  4. 手动清空了缓存 openSession.clearCache()；

二级缓存

  • 二级缓存（second level cache），全局作用域缓存，namespace 级别；

  • 二级缓存默认不开启，需要手动配置；

  • MyBatis 提供二级缓存的接口以及实现，缓存实现要求 POJO 实现 Serializable 接口；

  • 二级缓存在 SqlSession 关闭或提交之后才会生效。

要启用全局的二级缓存，只需要在你的 SQL 映射文件中添加一行：（当然，需要在全局配置文件中开启二级缓存 <setting name="cacheEnabled" value="true">）

这个语句的效果如下:

  • 映射语句文件中的所有 select 语句的结果将会被缓存；
  • 映射语句文件中的所有 insert、update 和 delete 语句会刷新缓存；
  • 缓存会使用最近最少使用算法（LRU, Least Recently Used）算法来清除不需要的缓存；
  • 缓存不会定时进行刷新（也就是说，没有刷新间隔）；
  • 缓存会保存列表或对象（无论查询方法返回哪种）的 1024 个引用；
  • 缓存会被视为读/写缓存，这意味着获取到的对象并不是共享的，可以安全地被调用者修改，而不干扰其他调用者或线程所做的潜在修改。

注意：缓存只作用于 cache 标签所在的映射文件中的语句。

这些属性可以通过 cache 元素的属性来修改。比如：

上面的配置创建了一个 FIFO 缓存，每隔 60 秒刷新，最多可以存储结果对象或列表的 512 个引用，而且返回的对象被认为是只读的，因此对它们进行修改可能会在不同线程中的调用者产生冲突。

  • eviction 清除策略：
    • LRU：最近最少使用，移除最长时间不被使用的对象，默认的；
    • FIFO：先进先出，按对象进入缓存的顺序来移除它们；
    • SOFT：软引用，基于垃圾回收器状态和软引用规则移除对象；
    • WEAK：弱引用，更积极地基于垃圾收集器状态和弱引用规则移除对象。
  • flushInterval：刷新间隔。属性可以被设置为任意的正整数，设置的值应该是一个以毫秒为单位的合理时间量。 默认情况是不设置，缓存仅仅会在调用语句时刷新；
  • size：引用数目。属性可以被设置为任意正整数，要注意欲缓存对象的大小和运行环境中可用的内存资源。默认值是 1024。
  • readOnly：只读，属性可以被设置为 true 或 false。只读的缓存会给所有调用者返回缓存对象的相同实例。 因此这些对象不能被修改。这就提供了可观的性能提升。而可读写的缓存会（通过序列化）返回缓存对象的拷贝。 速度上会慢一些，但是更安全，因此默认值是 false。

提示：二级缓存是事务性的。

整合第三方缓存

以 ehcahe 为例：

  1. 导包：
    1. ehcache-core-XXX.jar，ehche核心包；
    2. mybatis-ehcache-xxx.jar，mybatis 整合 ehcache 的整合包，也可以不导，但要自己实现mybatis 的 Cache 接口；
    3. slf4j-api-xxx.jar和slf4j-log4-xxx.jar：日志包；
  2. ehcache 要工作要有一个配置文件：文件名叫 ehcache.xml，放在类路径的根目录下；
  3. 在 mapper.xml 中配置使用自定义缓存：<cache type="org.mybatis.caches.ehcache.EhcacheCache"/> ；
  4. 别的 dao 还要用这个缓存，则可以在该 dao 的 mapper.xml 中配置<cache-ref namespace=""/>来和别的 dao 共用一个缓存。

注意事项

  1. 不会出现一级缓存和二级缓存中有同一个数据的情况；
    1. 二级缓存中，一级缓存关闭了就没有了；
    2. 一级缓存中，二级缓存中没有此数据，就会看一级缓存，一级缓存没有就去看数据库，数据库的查询结果放在一级缓存中；
    3. SqlSession 关闭，就会把一级缓存的数据放在二级缓存中；
  2. 任何时候都是先看二级缓存，再看一级缓存。

缓存有关设置

  1. 全局 setting 的 cacheEnable：配置二级缓存的开关。一级缓存一直是打开的；
  2. select 标签的 useCache 属性：配置这个 select 是否使用二级缓存。一级缓存一直是使用的；
  3. sql 标签的 flushCache 属性∶
    • 增删改默认 flushCache=true。sql 执行以后，会同时清空一级和二级缓存；
    • 查询默认 flushCache=false；
  4. sqlSession.clearCache() ：只是用来清除一级缓存；
  5. 当在某一个作用域(一级缓存Session/二级缓存Namespaces)进行了CUD操作后，默认该作用域下所有select中的缓存将被clear。

动态SQL

动态 SQL 是 MyBatis 强大特性之一，极大的简化我们拼装SQL的操作。

动态 SQL 元素和使用 JSTL 或其他类似基于 XML 的文本处理器相似。

MyBatis 采用功能强大的基于 OGNL 的表达式来简化操作。

主要的一些标签：

  • if
  • choose(when, otherwise)
  • trim (where, set)
  • foreach

更多请参考详细的官方文档

if 判断

支持OGNL表达式。

在 MyBatis 中，不只传入的参数可以用来做判断，额外还有两个东西：

  • _parameter：代表传入的参数；
    • 传入了单个参数：_parameter 就代表这个参数；
    • 传入了多个参数：_parameter 就代表多个参数集合起来的map；
  • _databaseId：代表当前环境；
    • 如果配置了 databaseIdProvider：_databaseId 就有值。

trim (where, set)

where

当 if判断 的示例中，我们 id 为 null 的时候，就会出现问题，拼成的字符串会表成 where and teacherName like...

而我们可以使用 where 标签来解决它（需要注意的是，and 要在要拼串的前面，在后面也不行。还有就是用了 where 标签后，在 sql 中就**不用写 where **了，因为where标签替代了它。）

trim

where 标签只会在子元素返回有任何内容的情况下才插入 “WHERE” 子句。而且，若子句的开头为 “AND” 或 “OR”，where 元素也会将它们去除。

  • prefix：为我们后面的sql整体添加一个前缀；
  • prefixOverrides：去掉后面整体字符串开头(指定的)多余的字符；
  • sufix：为整体添加一个后缀；
  • sufixOverrides：去掉后面整体字符串末尾(指定的)多余的字符。

set

用于动态更新语句的类似解决方案叫做 set。set 标签可以用于动态包含需要更新的列，忽略其它不更新的列。

foreach

动态 SQL 的另一个常见使用场景是对集合进行遍历（尤其是在构建 IN 条件语句的时候）。

  • foreach 元素的功能非常强大；
    • 它允许你指定一个集合，声明可以在元素体内使用的集合项（item）和索引（index）变量；
    • 它也允许你指定开头与结尾的字符串以及集合项迭代之间的分隔符。这个元素也不会错误地添加多余的分隔符。
  • 你可以将任何可迭代对象（如 List、Set 等）、Map 对象或者数组对象作为集合参数传递给 foreach；
    • 当使用可迭代对象或者数组时，index 是当前迭代的序号，item 的值是本次迭代获取到的元素；
    • 当使用Map对象（或者 Map.Entry 对象的集合）时，index 是键，item 是值。

choose、when、otherwise

有时候，我们不想使用所有的条件，而只是想从多个条件中选择一个使用。
