Git笔记

发表于2020-12-28|更新于2020-12-28|Git
|字数总计:3.4k|阅读时长:11分钟|阅读量:|评论数:

Git是一个免费的、开源的分布式版本控制系统，旨在快速高效地处理从小型到大型的所有项目。

Git 基本介绍

Git 官网

安装

对于Window操作系统，直接去官网下载安装包，然后直接安装即可。

配置用户

命令：

  • git config --global user.name 'myname'
  • git config --global user.email 'email@example.com'
  • git config --list

–global 表示是系统的用户级别的配置；

–system 是系统级别的配置；

不写表示是项目级别的配置。

初始化本地仓库

命令：git init

使用命令后，目录中会出现一个.git的隐藏文件夹，文件夹中有以下内容：

文件名 作用
hooks 目录包含客户端或服务端的钩子脚本。
info 包含一个个全局性排除文件。
logs 保存日志信息。
objects 目录存储所有数据内容。
refs 目录存储指向数据（分支）的提交对象指针。
config 文件包含项目特有的配置选项。
description 用来显示对仓库的描述信息。
HEAD 文件只是目前被检出的分支。
index 文件保存暂存区信息。

Git 中的区域

工作区、暂存区、版本库

Git 中的对象

这部分可能比较底层，本人的笔记也可能不够详尽，可以选择跳过。如果有兴趣可以去尚硅谷Git教程全套完整版（12h深入掌握git）看看，顺带提一句本人的笔记就是根据该视频整理。

  • Git 对象

    • 概念：Git 的核心部分是一个简单的键值对数据库。键值在 git 内部是 blob 类型数据；

      • 例如：向数据库写入内容并返回对应的键值；

      • 命令：echo 'test content' | git hash-object -w --stdin；

        • -w 表示 hash-object 命令存储数据对象；若不指定此选项，则该命令仅返回对应的键值；
        • –stdin (standerad input) 选项则指示改命令从标准输入读取内容；若不指定此选项，则必须在目录尾部给出待存储文件的路径。

      • 命令：git hash-objcet -w 文件路径；

        • 用于存文件。

      • 命令：git hash-object 文件路径；

        • 返回对应文件的键值 。

      • 命令：git cat-file -p 70460b4b4aece5915caf5c68d12f560a9fe3e4；

        • 根据键值拉去数据；
        • -p 选项可指示该命令自动判断内容的类型，并为我们显示格式友好的内容；
        • 返回：对应文件的内容。

    • 问题：我们每次修改文件里的内容，并存入数据库都会生成一个SHA-1值；

      • 记住文件的每一个版本所对应的SHA-1值并不现实；
      • 在Git中，文件名并没有被保存-我们仅保存了文件的内容。
        • 解决方案：数对象。

    • 注意：当前的操作都是在对本地数据库进行操作，不涉及暂存区。

  • 树对象

    • 树对象 (tree object)，它能解决文件名保存的问题，也允许我们将多个文件组织到一起。Git 以一种类似于 UNIX 文件系统的方式存储内容，所有内容均以树对象和数据对象 (git对象) 的形式存储，其中树对象对应了 UNIX 中的目录项，数据对象 (git对象) 则大致上对应文件内容。一个树对象包含了一条或多条记录（每条记录含有一个指向git对象或者子树对象的SHA-1指针，以及相应的模式、类型、文件名信息）。一个树对象也可以包含另一个树对象。
    • 查看暂存区当前样子：git ls-files -s；
    • 查看树对象——命令：git cat-file -p master^{tree}{或者是树对象的hash}。
      • master^{tree}语法表示master分支上最新的提交所指向的树对象。
    • 构建树对象：
      1. 利用 update-index 命令为test.txt文件的首个版本创建一个暂存区。并通过write命令生成树对象：
        • git update-index --add --cacheinfo 100644 70460b4b4aece5915caf5c68d12f560a9fe3e4 文件名
        • git write-tree
          • 文件模式：
            • 100644，表明这是一个普通文件；
            • 100755，表示一个可执行文件；
            • 120000，表示一个符号链接。
          • –add 选项：因为此前该文件并不在暂存区中，第一次需要-add；
          • –cacheinfo 选项：因为将要添加的文件位于Git数据库中，而不是位于当前目录下所有需要-cacheinfo。
      2. 新增new.txt将和text.txt文件的第二个版本new.txt塞入暂存区。并通过write-tree命令生成对象：
        • echo 'new filr' > new.txt
        • git update-index --cacheinfo 100644 70460b4b4aece5915caf5c68d12f560a9fe3e4 文件名
        • git updata-index --add new.txt
      3. 将第一个树对象加入第二个树对象，使其成为新的对象，命令：
        • git read-tree --prefix=bak 70460b4b4aece5915caf5c68d12f560a9fe3e4{第一个树对象的SHA-1}
        • git write-tree
          • read-tree 命令，可以把树对象读入暂存区。

  • 提交对象

    • 我们可以通过调用 commit-tree 命令创建一个提交对象，为此需要指定一个树对象SHA-1值，以及该提示的父提交对象（如果有的话第一次将暂存区做快照就没有父对象）;
    • 创建提交对象：echo "first commit" | git commit-tree SHA-1 | SHA-1父提交对象；
    • 查看提交对象：git cat-file -p SHA-1。

Git 本地操作命令

Git操作最基本的流程

  • 创建工作目录 => 对工作目录修改
  • git add ./
    • 等同于：
      • git hash-object 文件名 (修改了多少个工作目录中的文件，此命令就要被执行多少次)
      • git update-index ....
  • git commit -m "注释内容"
    • 等同于：
      • git write-tree
      • git commit-tree

git高层命令

  • git init：初始化目录；

    • 作用：初始化后，在当前目录下会出现一个名为.git的目录，所有Git需要的数据和资源都存放在这个目录中；
    • 注意：此时仅仅是按照既有的结构框架初始化好了里面所有的文件和目录，但我们还没有开始跟踪管理项目中的任何一个文件。

  • git add ./：将当前目录下所有修改添加到暂存区；

  • git add 文件名：跟踪一个新文件；

  • git commit -m 注释：将暂存区提交到版本库；

  • git commit：将会进入vim编辑模式，用来写注释；

  • git commit -a：跳过暂存区提交已经被跟踪的文件到版本库；

  • git status：查看当前文件状态（已提交、已修改或已暂存）；

  • git diff：查看当前有哪些更新还没有暂存；

  • git diff -cached 或者 git diff -staged(1.6.1以上)：查看有哪些更新已经暂存起来准备好下次提交

  • git mv file newfilename：改文件名；

    • 等同于：执行了一下三条命令。
      • mv README.txt README
      • git rm README.txt
      • git add README

  • git log：查看提交的历史记录。

    • git log --pretty=oneline
    • git log --oneline

Git 分支操作命令

  • git branch：显示当前所有分支列表；

  • git branch 分支名：为你创建一个可以移动的新指针；

  • git checkout 分支名：切换到某个分支；

  • git checkkout -b 文件名：新建一个分支，并切换到新建的分支；

  • git branch -d 分支名：删除分支，注意：不能再本分支下删除自己；

  • git branch -D 分支名：强制删除，因为-d不能删除未被合并的分支；

  • git branch -v：查看每一个分支最后一次的之后一次提交；

  • git branch name commitHas：新建一个分支并使分支指向新的提交对象；

  • git -log --oneline --decorate --graph --all：查看项目分叉历史，即查看项目所有的提交，包括其它分支的提交；

  • git config --global alias.co checkout：配置别名，本例中配置好后，就可以在命令中用co来代替checkout；

  • git merge 分支名：用当前分支去合并指定的分支。

Git 撤销&重置命令

Git 存储

当我们再项目的一部分上工作了一段时间后，所有东西都进入了混乱状态，而这时你想要切换到另一个分支做一点别的事情。问题是，你不想仅仅因为过会儿回到这一点而做了一半的工作创建一次提交。这时，我们可以使用 git stash。

命令：

  • git stash：命令会将未完成的修改保存到一个栈上，而你可以再任何时候重新应用这些改动 (git stash apply)；

  • git stash list：查看存储；

  • git stash apply stash@{2}：如果不指定一个存储，Git认为指定的是最近的存储；

  • git stash pop：来应用储藏然后立即从栈上扔掉它；

  • git stash drop stash@{2}：加上将要移除的储藏的名字来移除它。

后悔药（撤销&重置）

工作区：

  • 如何撤回自己在工作目录中的修改： git checkout --filename；

暂存区：

  • 如何撤回自己的暂存: git reset HEAD filename；

版本库：

  • 提交时注释写错了，改注释：git commit --amend；

reset

git log：查看提交的历史记录；

git reflog：只要是HEAD有变化，那么git reflog就会记下来。

重置三部曲：

  • 第一部： git reset -soft HEAD~

    • 这与移动HEAD自身不同 (checkout所做的)，reset 移动 HEAD 指向的分支；
    • 它本质上是撤销了上一次 git commit 命令；
    • HEAD~ 是指HEAD的父节点；
    • 可以把 HEAD~ 替换成一个提交对象，来撤回撤回，也可以实现多级撤回。

  • 第二部：git reset --mixed HEAD~

    • 带动HEAD（带着分支一起移动）；
    • 还取消了暂存区中的所有内容；
    • –mixed 也可以不写，因为默认就是–mixed。

  • 第三部：git reset --hard HEAD~

    • 带动HEAD（带着分支）；
    • 取消了暂存区中的所有内容；
    • 还回退了工作目录中的内容/

注意：–hard 是标记 reset 命令唯一的危险用法，它也是Git会真正地销毁数据的仅有几个操作之一。

路径reset：

  • git reset [--mixed] HEAD filename：(reset将会跳过第一步，也就是HEAD带着分支移动)；
  • 只动暂存区中指定的文件。

Git 数据恢复

在我们使用Git的时候，可能会意外丢失一次提交。通常这是因为我们强制删除了正在工作的分支，但是最后却发现我们还需要这个分支，以后这硬重置了一个分支，放弃了你想要的提交。那么，我们该如何找回提交？

最方便也是最常用的方法就是，使用git reflog命令。

  • 当我们在工作时，Git会默默记录每次你改变HEAD时它的值，每一次你提交或改变分支，引用日志都会更新，以此来找到我们想要回复的提交；
  • 有时候 git reflog 并不能显示足够的信息，所以，我们可以使用 git log -g，这个命令会以标准日志的格式输出引用日志；
  • 找到了想要恢复的提交，然后就可以使用 git branch recover-branch 提交对象（例如 git branch recover-branch ab1afe1），创建一个名为 recover-branch 的分支指向这个提交(abqafe1)。

Git 打tag

Git 可以给历史中的某一个提交打上标签，以示重要。比较有代表性的是人们会使用这个功能来标记发布结点(v1.0等等)。

相关操作：

  • 列出标签

    • git tag
    • git tag -l v1.8.5*

  • 创建标签

    • Git使用两种主要类型的标签：轻量标签和附注标签；
      • 轻量标签很像一个不会改变的分支。它指示一个特定提交的引用：
        • git tag v1.4
        • git tag v1.4 commitHash
      • 附注标签是存储在Git数据中的一个完整对象，它们是可以被校验的，其中包含打标签者的名字、电子邮箱地址、日期时间，还有标签信息，通常建议创建辅助标签，这样你可以拥有以上信息，但是如果你指示想用一个临时的标签，或者因为某些原因不想要保存那些信息，轻量标签也是可用的。
        • git tag -a v1.4
        • git tag -a v1.4 commitHash
        • git tag -a v1.4 commitHash -m 'my version 1.4'

  • 查看特定标签

    • git show 可以显示任意类型的对象(git 对象， 树对象 提交对象 tag对象)。
      • git show tagname

  • 远程标签

    • 默认情况下，git push 命令并不会传送标签到远程仓库服务器上。在创建完你标签后你必须显式的推送标签到共享服务器上。
      • git push origin [tagname]
      • 如果想要一次性推送很多标签，也可以使用带有 –tags 选项的 git push 命令，这将会把所有不在远程仓库服务器上的标签全部传送到那里；
      • git push origin --tags

  • 删除标签

    • 删除本地仓库上的标签：
      • git tag -d v1.4
      • 但是，上诉命令并不会删除远程仓库的这个标签，你必须使用g it push <remote>:refs/tags/<tagname> 来更新你的远程仓库。

  • 检出标签

    • 如果你想查看某个标签所指向的文件版本，可以使用 git checkkout 命令：
      • git checkout tagname
      • 虽说这会使你的仓库处于 “分离头指针” 状态。在 “分离头指针” 状态下，如果你做了某些更改然后提交它们，标签就不会发生变化，但你的新提交将不属于任何分支，并且将无法访问，除非访问确切的提交 hash。所以可以使用 git checkkout -b version2 来创建一个新的分支.
