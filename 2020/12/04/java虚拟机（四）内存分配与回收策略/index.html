java虚拟机（四）内存分配与回收策略

发表于2020-12-04|更新于2020-12-14|JavaJava虚拟机
|字数总计:2k|阅读时长:7分钟|阅读量:|评论数:

对象内存的分配主要是在堆上进行分配，对象主要会被分配在新生代的Eden区上，如果启动了本线程分配缓冲，将按线程优先在TLAB上分配。分配规则并非百分百固定的，其具体细节取决于当前使用的是哪一种收集器组合，以及虚拟机中的相关参数设置。

Minor GC（新生代GC）：指发生在新生代的垃圾收集动作，因为Java对象大多都具备朝生夕死的特性，所以Minor GC非常频繁，一般收回速度也比较快。

Major GC/Full GC（老年代GC）：指发生在老年代的GC。实现了Major GC，经常会伴随至少一次的Minor GC（并非绝对，在ParallelScavenge收集器的收集策略里，就直接进行Major GC）。Major GC 的速度一般都会比Minor GC慢10倍以上。

大对象直接进入老年代

大对象：指的就是需要大量连续内存空间的Java对象 。例如：那种很长的字符串及数组之类的。

经常出现大对象会提前触发垃圾收集以获取足够的连续空间来分配给大对象。

虚拟机提供了-XX:PretenureSizeThreshold参数，大于此值的对象直接在老年代分配，避免在 Eden 和 Survivor 之间的大量内存复制。

PretenureSizeThreshold只对Serial和ParNew两个收集器有效，而ParallelScavenge则是根据对象的大小是否小于Eden区的一半来判断（在jdk1.8中）。

长期存活的对象进入老年代

虚拟机为每个对象定义了一个对象年龄(Age)计数器。

如果对象在Eden区域出生经过第一次MinorGC后仍然存活，并且能够被Survivor容纳，那么就后被移动到Survivor空间中，并将器对象年龄设为1。

Survivor中的对象每渡过一次Minor GC它的年龄就会加1。当器年龄加到一定程度的时候，就会被晋升到老年代中。

对象晋升老年代的阈值，可以通过参数-XX:MaxTenuringThreshol来设置。

动态对象年龄判断

虚拟机并不是永远要求对象的年龄必须达到 MaxTenuringThreshold 才能晋升老年代。

如果在 Survivor 中相同年龄所有对象大小的总和大于 Survivor 空间的一半，则年龄大于或等于该年龄的对象可以直接进入老年代，无需等到 MaxTenuringThreshold 中要求的年龄。

空间分配担保

为什么需要空间分配担保？如果Survivor区出现有大量对象存活(甚至极端情况下所有对象都存活)的情况，那么就需要老年代进行分配担保，把Survivor无法容纳的对象直接进入到老年代中。

在发生Minor GC前，JVM会先检查老年代最大可用连续空间是否大于新生代所有对象的空间或者历次晋升的平均大小。如果大于，则说明空间足够，直接Minor GC；如果小于，则先进行一次Full GC。

JDK 6 Update 24前会根据HandlePromotionFailure参数判断是否允许担保失败；

  • 如果允许那么就会继续检查老年代最大可用的连续空间是否大于历次晋升到老年代对象的平均大小，如果大于，将尝试着进行一次 Minor GC；如果担保失败（Handle Promotion Failure）。则重新发起一次Full GC。
  • 如果不允许，那么就直接进行一次 Full GC。

老年代空间不足

老年代空间不足的常见场景为前文所讲的大对象直接进入老年代、长期存活的对象进入老年代等。

空间分配担保失败

使用复制算法的 Minor GC 需要老年代的内存空间作担保，如果担保失败会执行一次 Full GC。

JDK1.7及以前的永久代空间不足

在 JDK 1.7 及以前，HotSpot 虚拟机中的方法区是用永久代实现的，永久代中存放的为一些 Class 的信息、常量、静态变量等数据。

当系统中要加载的类、反射的类和调用的方法较多时，永久代可能会被占满，在未配置为采用 CMS GC 的情况下也会执行 Full GC。如果经过 Full GC 仍然回收不了，那么虚拟机会抛出 java.lang.OutOfMemoryError。

Concurrent Mode Failure

执行 CMS GC 的过程中同时有对象要放入老年代，而此时老年代空间不足（可能是 GC 过程中浮动垃圾过多导致暂时性的空间不足），便会报 Concurrent Mode Failure 错误，并触发 Full GC。
